Forced Browsing — How It Works

What is it?

Forced browsing (also called direct object browsing or path traversal of web roots) means accessing files and pages that exist on the server but are not linked in the application's navigation — because the server has no access control on them.

Discovery tools

# Gobuster — directory brute force
gobuster dir -u https://target.com -w /usr/share/wordlists/dirb/common.txt

# ffuf — fast fuzzer
ffuf -u https://target.com/FUZZ -w common.txt -mc 200,301,302

# Nikto — vulnerability scanner that checks for common files
nikto -h https://target.com

Most valuable targets

  • /.env — Contains DB passwords, API keys, JWT secrets
  • /config.json — Application credentials and settings
  • /backup/db.sql — Full database dump including password hashes
  • /.git/config — Reveals repository URL, may allow source dump
  • /admin/panel.html — Admin interface with no authentication
  • /uploads/ — May expose uploaded scripts, user data
  • /phpinfo.php — Server configuration, installed extensions
  • /server-status — Apache status page, active requests

The Fix

1. Move secrets out of the web root:
   - .env, config files → server environment variables
   - Database backups  → private storage, not /backup/

2. Enforce authentication on every non-public path:
   app.use("/admin", requireLogin);
   app.use("/api",   requireLogin);

3. Return 401/403 for unauthorised access — not 200.
   Returning 200 confirms the resource exists.

4. Disable directory listing in web server config:
   # Nginx:  autoindex off;
   # Apache: Options -Indexes

5. Use robots.txt only to guide crawlers — it is public!
   Listing /admin in robots.txt tells every attacker exactly where to look.
