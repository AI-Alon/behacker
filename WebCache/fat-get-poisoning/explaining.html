Fat GET Cache Poisoning

What is it?

A "fat GET" is a GET request with a request body. Some backend servers (particularly with certain frameworks or proxies) override query parameters with body parameters. If the CDN/cache only keys on the URL, the attacker can send a body-injected value that gets reflected, cached, and served to all users of that URL.

The desync

CDN cache sees:    GET /search?q=shoes   → caches this URL
Backend sees:      GET /search?q=shoes + body: q=<script>alert(1)</script>
                   → uses body param (overrides URL param)
                   → returns page with injected q
CDN caches:        <script>alert(1)</script> under key /search?q=shoes
Victim gets:       Clean URL /search?q=shoes → poisoned response

When servers accept GET bodies

  • Some PHP frameworks: $_POST takes precedence
  • Python requests: body is ignored for GET by default, but some servers accept it
  • Some Java servlet containers process body even for GET
  • Reverse proxies that strip/forward bodies differently

Detection

Use Burp Suite Param Miner or manual testing: send GET with body containing a unique canary value. If the canary appears in the response, the server processes GET bodies.

Defense

  • Configure CDN/cache to reject or normalize GET requests with bodies
  • Ensure backend framework ignores body parameters in GET requests
  • Add body content to cache keys if body processing is intentional
  • Use strict HTTP method semantics: GET must be idempotent and have no body side effects
