Web Cache Poisoning — Cache Key Manipulation

What is it?

Web caches store responses keyed on the request URL (and sometimes specific headers). If the server uses a header that is NOT part of the cache key to generate the response (e.g. reflecting X-Forwarded-Host into script tags), an attacker can poison the cached response. Every subsequent visitor who requests the same URL receives the malicious response.

Mechanism

Cache key:   GET /homepage
             Host: securecorp.com
             [X-Forwarded-Host excluded from key]

Attacker:    GET /homepage
             X-Forwarded-Host: evil.com

Server reflects: <script src="https://evil.com/cdn/app.js"></script>
Cache stores this response under key GET:/homepage

Victim:      GET /homepage  (no X-Forwarded-Host header)
             → Cache HIT → receives poisoned page → loads evil.js

Common unkeyed headers

  • X-Forwarded-Host, X-Forwarded-Scheme, X-Forwarded-Proto
  • X-Original-URL, X-Rewrite-URL
  • X-Forwarded-For (when reflected in responses)
  • Origin, User-Agent (sometimes used for feature detection)

Real-world examples

  • Portswigger research (2018): Dozens of major sites vulnerable to cache poisoning via X-Forwarded-Host
  • CPDoS (2019): Cache Poisoned Denial of Service — poison cache to serve 400 errors

Defense

# Include all response-affecting headers in cache key
add_header Vary "X-Forwarded-Host, X-Forwarded-Scheme";

# Or: don't reflect these headers in responses
# Validate X-Forwarded-Host against known values
  • Never reflect unvalidated headers into script/link tags
  • Use Vary to include headers that affect the response in the cache key
  • Run automated cache poisoning tests before deploying (Param Miner Burp extension)
