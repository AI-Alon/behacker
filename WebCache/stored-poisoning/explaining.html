Stored Web Cache Poisoning

What is it?

Stored cache poisoning is a variant where an attacker's single request poisons a shared cache entry, causing all future visitors to receive the malicious response — without the victim ever visiting the exploit URL. The attacker "stores" the XSS payload in the cache.

Why it's more powerful than reflected XSS

  • Reflected XSS: Victim must click attacker's link
  • Stored cache XSS: Victim visits any bookmarked URL → receives poisoned response from cache
  • No user interaction beyond visiting a legitimate URL

Attack flow

  1. Attacker finds a parameter reflected in the response that is NOT part of the cache key
  2. Attacker sends: GET /homepage?utm_content=</script><script>alert(document.cookie)</script>
  3. Server reflects it into: <script>var tracking='INJECTED_PAYLOAD';</script>
  4. Cache stores response under key GET /homepage (utm_content excluded from key)
  5. Every visitor to /homepage gets the XSS payload

Common reflection points targeted

  • Analytics parameters: utm<em>source, utm</em>content, fbclid
  • Language/locale parameters: lang=en
  • Callback parameters: cb=1234 (cache busters that get reflected)
  • Origin header reflected in CORS responses

Defense

  1. Strip or sanitize unrecognized query parameters before reflecting them
  2. Include parameters in cache key if they affect the response (using Vary or cache key rules)
  3. Output encoding: encode all reflected values as JavaScript strings properly
  4. CSP: A strict Content-Security-Policy limits the impact of XSS
