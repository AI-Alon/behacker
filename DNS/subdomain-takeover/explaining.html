Subdomain Takeover

What is it?

Subdomain takeover occurs when an organization removes a cloud resource (GitHub Pages, Heroku app, AWS S3 bucket, Netlify site) but leaves the DNS CNAME record pointing to it. An attacker can register the same resource name on that platform, gaining control of all traffic to that subdomain.

Step-by-step

  1. Organization sets: staging.example.com CNAME → example.github.io
  2. Organization deletes the GitHub Pages site (but forgets the DNS record)
  3. example.github.io is now unclaimed — anyone can create it
  4. Attacker creates a GitHub Pages repo named example
  5. staging.example.com now serves attacker content under the legitimate domain

Why it's severe

  • Trusted domain: The subdomain inherits the parent's reputation and TLS cert
  • Cookie scope: Cookies set for .example.com are readable from staging.example.com
  • CORS bypass: Origins like staging.example.com may be whitelisted in CORS policies
  • OAuth redirect: If staging.example.com is a registered OAuth redirect URI, attacker receives tokens
  • Email reputation: Subdomain can send phishing emails appearing to come from example.com

Vulnerable platforms (partial list)

GitHub Pages, Netlify, Heroku, AWS S3/CloudFront, Azure Web Apps, Fastly, Pantheon, Zendesk, UserVoice, Freshdesk, WP Engine

Detection tools

  • subjack — scans subdomain list for dangling CNAMEs
  • nuclei with takeover templates
  • can-i-take-over-xyz — GitHub repo of platform fingerprints
  • dnsX — fast DNS resolution and analysis

Prevention

  1. Remove DNS records before deprovisioning resources
  2. Manage DNS and cloud resources together in IaC (Terraform/Pulumi)
  3. Regular automated DNS audits against active resource inventory
  4. Use NS record monitoring to alert on unregistered CNAME targets
