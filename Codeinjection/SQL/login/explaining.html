SQL Injection — Login Bypass

What is it?

SQL injection in a login form allows an attacker to bypass authentication entirely by injecting SQL syntax that short-circuits the WHERE clause. The server never actually verifies the password — the injected comment discards that check.

How it works

Normal authentication query — checks username and password against the users table:

SELECT * FROM users WHERE username = 'input' AND passwd = 'input';

If we input the following values:

  • username: admin'--
  • passwd: anything

The resulting query will be:

SELECT * FROM users WHERE username = 'admin'--' AND passwd = 'anything';

The -- starts a comment, so the password check is ignored. The query effectively becomes:

SELECT * FROM users WHERE username = 'admin';

Result: logged in as admin with no password required.

The fix: use a parameterized query. The user input is passed as a bound value, never concatenated into the SQL text.

SELECT * FROM users WHERE username=? AND passwd=?
-- values: (username, passwd)

No matter what the user types, it is treated as plain data. A username of admin'-- becomes a literal string lookup — not SQL syntax. The comment characters have no special meaning inside a bound parameter.

Real-world impact

  • Authentication bypass: Log in as any user without knowing their password
  • Admin account compromise: Target well-known usernames like admin, root, superuser
  • Full database dump: Chain with UNION injection to extract all data
  • CVE-2011-4107: phpMyAdmin SQL injection bypass — classic example

Common payloads

admin'--          → comments out password check
' OR '1'='1'--   → makes WHERE always true
' OR 1=1--        → same, numeric comparison
admin'/*          → block comment variant
