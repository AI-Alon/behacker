SSTI (Server-Side Template Injection) — How It Works

Normal template rendering (intended)

The server accepts a template string and renders it with safe context variables:

template = "Hello {{name}}, your role is {{role}}."
render(template, name="Alice", role="Viewer")

Output: Hello Alice, your role is Viewer.

Injection: access variables outside the safe set

Input template: {{secret}}

The engine evaluates secret against the full server context — which includes

variables the developer never intended to expose.

Output: FLAG{ssti<em>template</em>pwned}

Injection: traverse nested objects

Input template: {{config.db_url}}

Output: postgres://admin:hunter2@db:5432/prod

The template engine follows dot-path traversal, reaching the database connection string.

Injection: loop to dump all users

Input template:

{% for u in users %}{{u.username}} — {{u.email}}
{% endfor %}

Output: full list of all user accounts and emails.

Escalation: Jinja2 Python object traversal (real server)

In a real Jinja2 environment, template expressions can walk Python's object hierarchy:

{{''.__class__.__mro__[1].__subclasses__()}}

This reaches Python's base object class and lists all subclasses currently loaded —

including file I/O, subprocess, and OS interfaces.

From there an attacker can execute arbitrary OS commands with no further authentication.

Why this is different from XSS

  • XSS runs in the browser — limited to what the page can do.
  • SSTI runs on the server — full access to the process, filesystem, and network.
  • SSTI can lead to Remote Code Execution (RCE) — the highest severity finding.

The Fix

# WRONG — user controls what gets evaluated
template_str = request.form["template"]
render_template_string(template_str, **context)

# RIGHT — user input goes into context values, not the template structure
render_template("fixed_email.html", name=request.form["name"])

The template structure is fixed and written by the developer.

User input is only ever a context value — it is never evaluated as template syntax.
