JS Injection — How It Works

Normal eval usage (already bad, but intended)

The app builds a greeting expression and evaluates it:

const code = `"Hello, " + "${name}" + "! Welcome."`;
eval(code);

If name = Alice, the code string is:

"Hello, " + "Alice" + "! Welcome."

Result: Hello, Alice! Welcome.

Injection: close the string, append code

Input name: " + SERVER_CONTEXT.secret + "

Resulting code string:

"Hello, " + "" + SERVER_CONTEXT.secret + "" + "! Welcome."

The injected expression reads the secret variable from scope and concatenates it into the output.

Result: Hello, FLAG{js<em>injection</em>pwned}! Welcome.

Escalation: arbitrary statement execution

Input: "; alert('injected'); //

Resulting code string:

"Hello, " + ""; alert('injected'); //" + "! Welcome."

alert() executes as a standalone statement. The // comments out the rest.

Why eval() is dangerous

  • It executes any JavaScript expression in the current scope.
  • It has access to all local and global variables, closures, and imported modules.
  • There is no sandboxing — it runs with full page privileges.

The Fix

Never use eval() with user input. Use safe DOM APIs instead:

// Safe: just set text
greetingEl.textContent = `Hello, ${sanitize(name)}! Welcome.`;

// If you need dynamic logic: use a whitelist or a safe template function
// — never eval a string that contains user-supplied content.
