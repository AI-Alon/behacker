Command Injection — How It Works

Normal command (intended behaviour)

The server takes a hostname and runs ping:

os.system("ping -c 4 " + hostname)

Input: example.com

Command executed: ping -c 4 example.com

Injection: semicolon separates commands

Input: example.com; whoami

Command executed:

ping -c 4 example.com; whoami

The shell runs ping first, then runs whoami as a second command.

Output includes the ping result followed by www-data.

Injection: && chains on success

Input: example.com && cat /etc/passwd

ping -c 4 example.com && cat /etc/passwd

&& runs the second command only if ping succeeds.

Output leaks the contents of /etc/passwd.

Injection: | pipes output

Input: example.com | cat .env

ping -c 4 example.com | cat .env

Pipes ping stdout into cat — but cat ignores stdin and reads the file.

Leaks environment secrets like DB passwords and API keys.

Shell metacharacters to know

  • ; — Run next command regardless of exit code
  • && — Run next command if previous succeeded
  • || — Run next command if previous failed
  • | — Pipe stdout of left into stdin of right
  • ` ` — Command substitution — execute and insert output
  • $() — Command substitution (modern form)

The Fix

import subprocess

# Safe: list form — no shell involved, metacharacters are literal
subprocess.run(["ping", "-c", "4", hostname])

# Also: validate hostname is a safe string before using it at all
import re
if not re.fullmatch(r'[\w.\-]+', hostname):
    raise ValueError("Invalid hostname")

When you pass a list instead of a string, Python never invokes a shell.

There is no shell parser, so metacharacters have no special meaning.
