CSRF — How It Works

What is it?

CSRF (Cross-Site Request Forgery) tricks a logged-in user's browser into sending a request to a site they are authenticated on — without their knowledge. The browser automatically attaches the victim's session cookie to every request, regardless of which site triggered it.

The attack flow

1. Victim logs into securecorp.com — session cookie stored in browser
2. Victim visits evil.com (malicious page)
3. evil.com contains a hidden auto-submitting form:

   <form action="https://securecorp.com/api/transfer" method="POST" id="x">
     <input name="to"     value="attacker-account">
     <input name="amount" value="500">
   </form>
   <script>document.getElementById('x').submit();</script>

4. Browser sends POST to securecorp.com
5. Browser auto-attaches: Cookie: session=alice_valid_session
6. Server sees a valid session — processes the transfer
7. $500 transferred to attacker, victim has no idea

Why cookies alone are not enough

The server cannot distinguish:

  • A request from securecorp.com/transfer-form (legitimate)
  • A request from evil.com hidden form (forged)

Both carry the same session cookie.

The Fix

CSRF Token (synchroniser token pattern)

<!-- Server embeds a secret token in every form -->
<form method="POST" action="/api/transfer">
  <input type="hidden" name="csrf_token" value="a3f9c2e1b847...">
  ...
</form>
// Server verifies the token on every state-changing request
if (req.body.csrf_token !== req.session.csrf_token) {
  return res.status(403).send("CSRF token mismatch");
}

The attacker's page cannot read the token (blocked by Same-Origin Policy), so it cannot forge a valid request.

SameSite Cookie Attribute

Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly

SameSite=Strict — cookie is never sent on cross-site requests.

SameSite=Lax — sent on top-level navigations (GET) but not cross-site POST.
