Clickjacking — How It Works

What is it?

Clickjacking tricks a victim into clicking a UI element they cannot see. The attacker embeds the target page in a transparent iframe and overlays it on a decoy page, aligning a dangerous button with something the victim wants to click.

How it works

Attacker's page (evil.com):
┌──────────────────────────────────┐
│  Win a Free iPhone!              │  ← decoy page (visible)
│  Click to claim your prize       │
│                                  │
│  [ Claim Prize ]                 │  ← victim clicks here
└──────────────────────────────────┘

Invisible iframe on top (opacity: 0):
┌──────────────────────────────────┐
│  My Account — Secure Corp        │  ← real target page (invisible)
│  Logged in as: alice             │
│                                  │
│  [ Delete Account ]              │  ← aligned under decoy button
└──────────────────────────────────┘

The victim's click lands on the invisible "Delete Account" button.

Requirements

  1. The target page allows iframe embedding (no X-Frame-Options or frame-ancestors)
  2. The dangerous action triggers on a single click (no confirmation dialog)
  3. The victim is logged into the target site in another tab

Common targets

  • Account deletion / password change
  • Bank transfers
  • OAuth permission grants ("Allow this app")
  • Social media likes / follows / shares (likejacking)
  • Admin actions (promote user, enable setting)

The Fix

# HTTP response header — most reliable
X-Frame-Options: DENY
# or to allow same-origin embedding only:
X-Frame-Options: SAMEORIGIN

# Content Security Policy (more flexible)
Content-Security-Policy: frame-ancestors 'none';
# or:
Content-Security-Policy: frame-ancestors 'self' https://trusted.com;

Add these headers to every page that performs sensitive actions.

frame-ancestors in CSP overrides X-Frame-Options in modern browsers.
