Stored XSS — How It Works

Normal rendering (safe)

Messages are saved and displayed as plain text:

messageEl.textContent = storedMessage;

Output: <script>alert(1)</script> ← shown literally, not executed.

Vulnerable rendering (stored + innerHTML)

// Step 1: save raw input
localStorage.setItem("messages", JSON.stringify([{ name, message }]));

// Step 2: on every page load, render stored HTML
boardOutput.innerHTML = storedMessages.map(m =>
  `<div><span>${m.name}</span><p>${m.message}</p></div>`
).join("");

If a message contains: <img src=x onerror="alert('XSS')">

That tag is stored, then replayed into the DOM on every page load — for every visitor.

Why "stored" is more dangerous than reflected

  • Payload location: Reflected XSS — URL parameter | Stored XSS — Database / storage
  • Who is affected: Reflected XSS — Only victims who click the crafted link | Stored XSS — Every user who visits the page
  • Server logs: Reflected XSS — Payload may appear in access logs | Stored XSS — Payload is already stored — no suspicious URL
  • Requires victim action: Reflected XSS — Click a link | Stored XSS — Just visit the page

Real-world targets

  • Comment sections
  • User profile bios
  • Product reviews
  • Support ticket systems
  • Chat messages

The Fix

// Option 1: escape before storing
function escapeHtml(str) {
  return str
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;");
}

// Option 2: use textContent when rendering (never innerHTML for user content)
nameEl.textContent  = m.name;
msgEl.textContent   = m.message;

// Option 3: sanitise with a library (DOMPurify)
boardOutput.innerHTML = DOMPurify.sanitize(html);

Always sanitise on output — even if input validation is in place.
