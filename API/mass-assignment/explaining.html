Mass Assignment

What is it?

Mass assignment is a vulnerability where an API automatically binds all fields from a request body to a data model object — including fields that should only be set server-side, like role, isAdmin, or balance.

How it works

Modern frameworks (Rails, Django REST, Laravel, Spring) offer convenient auto-binding that maps incoming JSON/form data directly to model attributes. Without an explicit allowlist of permitted fields, any field in the model becomes writable by the client.

Vulnerable code (Node/Express):

app.put('/api/users/:id', async (req, res) => {
  await User.update(req.params.id, req.body);  // binds ALL body fields
});

Attack payload:

{
  "name": "Alice",
  "email": "alice@example.com",
  "role": "admin",
  "isAdmin": true,
  "balance": 99999.99
}

Impact

  • Privilege escalation: Set role: "admin" to gain administrative access
  • Financial fraud: Set balance or credits to arbitrary values
  • Feature unlocking: Change subscription plan to paid tiers for free
  • Data integrity: Overwrite protected fields like verified, emailConfirmed

Real-world examples

  • GitHub (2012): Mass assignment allowed adding SSH keys to any organization
  • Rails apps: attr_accessible was introduced specifically due to this class of bugs
  • HackerOne disclosures: Multiple API mass assignment reports on major platforms

Defense

Always use an explicit field allowlist:

// Express + validation
const allowed = ['name', 'email', 'bio'];
const safe = Object.fromEntries(
  Object.entries(req.body).filter(([k]) => allowed.includes(k))
);
await User.update(req.params.id, safe);

// Rails
user.update(params.require(:user).permit(:name, :email, :bio))

Never permit: role, isAdmin, balance, plan, verified, passwordHash, or any field that should only change through a controlled server-side process.
