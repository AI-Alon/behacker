API Key Exposure in Frontend Bundles

What is it?

API keys and secrets embedded in JavaScript bundles are completely public — any user who visits the site can view them via browser DevTools (Sources, Network, Console). This is not a subtle vulnerability; it is as if the secret was printed on the homepage.

How it happens

  1. Build-time injection: CI/CD pipelines inject environment variables into the bundle (REACT<em>APP</em><em>, VUE<em>APP</em></em>, NEXT<em>PUBLIC</em>*)
  2. Developer shortcuts: Developers hardcode keys temporarily and commit them
  3. Source maps: .js.map files ship the original pre-minified source code, including comments with credentials
  4. Copy-paste from server code: Server-side config files copied into shared frontend config

What attackers look for

  • sk_live_* — Stripe secret key
  • AKIA* — AWS Access Key ID
  • AIzaSy* — Google Maps/Cloud API key
  • eyJ* — JWT tokens (base64 header starts with eyJ)
  • tok_*, key_* — Generic API tokens

Tools for automated detection

  • Browser: DevTools → Sources → Search (Ctrl+Shift+F) for secret, key, token, password
  • trufflehog: Scans git history and files for 700+ secret patterns
  • gitleaks: Pre-commit hook and CI scanner
  • semgrep: Static analysis with secret detection rules

Impact

  • Stripe secret key → make charges, issue refunds, access customer data
  • AWS keys → spin up EC2, access S3 buckets, exfiltrate data
  • Twilio auth token → send SMS from your number
  • Admin JWT → authenticate as any user

Defense

Rule: If the browser can see it, it's not a secret.

  • Only use publishable/public keys client-side (e.g., Stripe's pk<em>live</em><em>, not sk<em>live</em></em>)
  • Proxy all sensitive API calls through your backend — the browser calls your server, your server calls the third-party API using server-side credentials
  • Use short-lived, scoped tokens for client-initiated operations
  • Block *.map files in production or strip them from the build
  • Run secret scanning as a pre-commit hook and in CI
