GraphQL Introspection & Missing Authorization

What is it?

GraphQL introspection is a built-in feature that lets clients query the full API schema — all types, fields, queries, mutations, and their arguments. In production, this is an attacker's roadmap to every internal operation your API supports.

The two-part vulnerability

1. Introspection enabled in production

{ __schema { types { name fields { name description } } } }

This reveals:

  • Hidden admin queries (adminUsers, adminDumpDB, adminSetRole)
  • Internal field names (passwordHash, resetToken, ssn)
  • Developer comments in description fields marked "INTERNAL"
  • The full mutation schema including destructive operations

2. No authorization on resolvers

Even if introspection is disabled, the resolvers themselves must verify the caller. Without authorization:

{ adminUsers { id email passwordHash resetToken } }

Returns all user records to any unauthenticated caller.

Attack flow

  1. Send introspection query → discover adminDumpDB, adminSetRole, adminResetAll
  2. Call adminDumpDB → get S3 URL to full database backup
  3. Call adminSetRole(userId: "101", role: "admin") → escalate privileges
  4. Use admin token to access entire application

Tools

  • GraphiQL / Altair: Browser-based GraphQL IDEs that auto-run introspection
  • InQL (Burp extension): Parses schema, generates test queries for every operation
  • graphql-voyager: Visualizes schema as an interactive graph

Defense

// Apollo Server — disable introspection
const server = new ApolloServer({
  introspection: process.env.NODE_ENV !== 'production',
});

More importantly, authorization in every resolver:

adminUsers: (_, __, context) => {
  if (!context.user?.isAdmin) throw new ForbiddenError('Admins only');
  return User.findAll();
}
  • Disable introspection in production
  • Apply field-level authorization (directives or middleware)
  • Separate public and internal schemas if possible
  • Use query depth/complexity limits to prevent abuse
