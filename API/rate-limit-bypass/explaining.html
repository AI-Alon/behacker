Rate Limit Bypass via Header Spoofing

What is it?

Many APIs implement rate limiting based on the client's IP address. But if the server trusts IP-override headers like X-Forwarded-For or X-Real-IP that any client can freely set, an attacker can simply rotate these values to bypass the limit entirely.

How it works

Legitimate use case: a load balancer or reverse proxy sets X-Forwarded-For so the app knows the real client IP behind the proxy.

Vulnerable code:

const clientIP = req.headers['x-forwarded-for'] || req.ip;
if (rateLimitExceeded(clientIP)) return res.status(429).send('Too Many Requests');

Attack:

POST /api/verify-otp
X-Forwarded-For: 10.0.0.1    → blocked after 3 attempts
X-Forwarded-For: 10.0.0.2    → new "IP", 3 more attempts
X-Forwarded-For: 10.0.0.3    → 3 more...

Headers commonly abused for bypass

  • X-Forwarded-For
  • X-Real-IP
  • X-Originating-IP
  • X-Remote-IP
  • CF-Connecting-IP
  • True-Client-IP
  • Forwarded: for=1.2.3.4

Impact

  • OTP / 2FA brute-force (4-digit OTP = 10,000 combinations)
  • Password reset token brute-force
  • Login credential stuffing without lockout
  • Enumeration of user accounts via response timing

Real-world examples

  • Multiple bug bounty reports on HackerOne for OTP bypass via header rotation
  • SMS 2FA bypassed on social platforms by cycling X-Forwarded-For

Defense

  1. Rate-limit on account identity, not just IP — lock the account after N failures regardless of source IP
  2. Validate proxy headers: Only trust X-Forwarded-For from known, configured upstream proxy IPs; reject it from untrusted sources
  3. Add delays and jitter between OTP attempts (even with lockout)
  4. CAPTCHA after the first failed attempt for sensitive actions
  5. Longer OTPs: 6+ digits, or use TOTP (time-based) which has a narrow validity window
