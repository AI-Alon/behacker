JWT Attacks — How It Works

JWT structure

A JWT is three base64url-encoded parts separated by dots:

HEADER.PAYLOAD.SIGNATURE

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.eyJzdWIiOiIxMDEiLCJ1c2VybmFtZSI6ImFsaWNlIiwicm9sZSI6InVzZXIifQ
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Anyone can decode the header and payload — they are not encrypted.

Security depends entirely on the signature being unforged.

Attack 1 — alg:none

Some JWT libraries read the alg field from the token's own header and use it for verification.

// Attacker changes header to:
{ "alg": "none", "typ": "JWT" }

// Removes the signature:
eyJhbGciOiJub25lIn0.eyJyb2xlIjoiYWRtaW4ifQ.

If the server accepts alg:none, it skips signature verification entirely.

Any claims in the payload are trusted — no secret needed.

Fix: hardcode the expected algorithm server-side, never read it from the token.

Attack 2 — Brute-force weak secret (HS256)

HS256 tokens can be verified offline — the attacker doesn't need to send requests to the server.

# hashcat
hashcat -a 0 -m 16500 token.jwt rockyou.txt

# jwt-cracker (Node)
jwt-cracker "eyJ..." wordlist.txt

Once the secret is known, the attacker can sign any payload they want.

Fix: use a random secret of at least 256 bits. Consider RS256 (asymmetric) so the signing key never leaves the server.

Attack 3 — Claim modification

Even without cracking the secret, if the attacker can resign (e.g., after brute-forcing), they modify:

{ "role": "admin", "sub": "1", "username": "alice" }

The server grants admin access because it trusts the role claim in the JWT.

Fix: treat JWT claims as hints only. Verify role and permissions against a server-side database on sensitive operations.

Common JWT weaknesses summary

  • alg:none accepted: Attack — Strip signature, forge claims | Fix — Hardcode expected alg server-side
  • Weak HMAC secret: Attack — Offline brute-force, re-sign | Fix — 256-bit random secret or RS256
  • No expiry (exp): Attack — Token valid forever | Fix — Set short exp, implement revocation
  • Sensitive data in payload: Attack — Decode and read | Fix — JWTs are not encrypted — don't store secrets
  • Role stored in JWT: Attack — Modify role claim | Fix — Re-validate role from DB on each request
