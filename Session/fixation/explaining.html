Session Fixation — How It Works

Normal session flow (safe)

1. User visits /login         → server creates session SID=RANDOM_A
2. User submits credentials   → server destroys RANDOM_A, creates new SID=RANDOM_B
3. Browser stores RANDOM_B    → only the server and this browser know it

The session ID is regenerated after login — an attacker who knew the pre-login SID can't use it.

Vulnerable flow (no regeneration)

1. Server accepts ?sid= from URL → pre-creates session SID=ATTACKER_KNOWN
2. Attacker sends victim: https://site.com/login?sid=ATTACKER_KNOWN
3. Victim logs in               → server binds alice's identity to SID=ATTACKER_KNOWN
4. Attacker sends:  Cookie: sid=ATTACKER_KNOWN
5. Server responds: Welcome, alice!

The attacker "fixed" the session ID before login — the server never replaced it.

Attack delivery methods

# URL parameter (most common)
https://site.com/login?sid=KNOWN123

# Meta refresh / iframe (no user interaction needed)
<meta http-equiv="refresh" content="0; url=https://site.com/login?sid=KNOWN123">

# Some servers accept SID in a hidden form field or cookie header

The Fix

// Express / Node example
app.post("/login", (req, res) => {
  if (validCredentials(req.body)) {
    // SAFE: destroy old session, regenerate ID
    req.session.regenerate((err) => {
      req.session.user = authenticatedUser;
      res.redirect("/dashboard");
    });
  }
});

Always regenerate the session ID immediately after a successful login.

Never accept a session ID from user-supplied input (URL, form field).
