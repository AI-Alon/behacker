Cookie Tampering — How It Works

What is stored in the cookie

The server sets a cookie containing the session data as base64-encoded JSON:

session=eyJ1c2VybmFtZSI6ImFsaWNlIiwicm9sZSI6InVzZXIiLCJ1aWQiOjEwMX0=

Decoded:

{ "username": "alice", "role": "user", "uid": 101 }

Why base64 is not security

Base64 is encoding, not encryption or signing.

atob("eyJ1c2VybmFtZSI6ImFsaWNlIiwicm9sZSI6InVzZXIiLCJ1aWQiOjEwMX0=")
// → '{"username":"alice","role":"user","uid":101}'

Anyone can decode it, modify it, and re-encode it — the server cannot tell.

The attack

1. Login as alice (role: user)
2. Decode cookie → {"username":"alice","role":"user","uid":101}
3. Modify        → {"username":"alice","role":"admin","uid":1}
4. Re-encode     → btoa(JSON.stringify(modified))
5. Replace cookie value in browser DevTools → Application → Cookies
6. Reload page   → server reads role:"admin" and grants admin access

Privilege escalation via uid

Changing uid to another user's ID is an IDOR (Insecure Direct Object Reference) attack:

{ "username": "alice", "role": "user", "uid": 102 }

The server now returns bob's data — no credential needed.

The Fix

// Option 1: HMAC-sign the cookie (cookie-signature library)
const sig  = crypto.createHmac("sha256", SECRET).update(value).digest("base64");
const safe = `${value}.${sig}`;
// On read: verify sig before trusting value

// Option 2: Store nothing sensitive in the cookie
// Cookie holds only an opaque session ID:
//   session=a3f9c2e1b847...
// All data (role, uid) lives in the server-side session store.
// Tampering with the ID produces a miss → no session → logged out.
