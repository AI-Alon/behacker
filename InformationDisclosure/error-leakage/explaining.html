Error Message Leakage — How It Works

Normal error response (safe)

HTTP/1.1 500 Internal Server Error

{ "error": "An unexpected error occurred. Please try again." }

The client gets a generic message. Details are logged server-side only.

Verbose error response (vulnerable)

HTTP/1.1 500 Internal Server Error

QueryError: Unknown column 'nonexistent_column' in 'order clause'
    at /var/www/securecorp/app/routes/products.js:47 (searchProducts)
    at /var/www/securecorp/app/middleware/validate.js:23 (runQuery)
    at /var/www/securecorp/app/node_modules/mysql2/lib/connection.js:130

Debug info (DEBUG=true):
  Framework:  Express 4.18.2
  Runtime:    Node.js v18.12.1
  Database:   MySQL 8.0.31 @ localhost (user: root, db: securecorp_prod)
  App path:   /var/www/securecorp/app
  Query:      SELECT * FROM products WHERE name LIKE '%laptop%' ORDER BY `nonexistent_column` LIMIT 10

What each leak reveals

  • Internal file paths: Target path traversal attacks at known locations
  • Framework + version: Look up CVEs for Express 4.18.2
  • DB type + version: Craft version-specific SQL injection payloads
  • DB username + host: Aid credential attacks and SSRF
  • SQL query structure: Understand table/column names for SQLi
  • Stack frame function names: Understand app logic, find injection points

Triggering errors intentionally

# Type errors — reveals parameter names and expected types
?limit=abc

# SQL errors — reveals query structure and column names
?sort=nonexistent_field

# Boundary errors — reveals limits and memory constraints
?limit=-1
?limit=999999999

# Injection probes — reveals whether input reaches SQL
?q='; DROP TABLE--

The Fix

// Express — production error handler
app.use((err, req, res, next) => {
  // Log full details server-side only
  console.error(err);

  // Return generic message to client
  res.status(500).json({ error: "Internal server error" });
});

// Environment-based config
if (process.env.NODE_ENV === "production") {
  app.set("debug", false);
}
# Never run production with DEBUG=true or NODE_ENV=development
NODE_ENV=production node app.js

Treat error messages as a privileged internal resource — never expose them to clients in production.
