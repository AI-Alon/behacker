Source Code Exposure

What is it?

Source code exposure occurs when web servers inadvertently serve files that should never be publicly accessible — version control directories (.git/), environment files (.env), backup files (<em>.bak), source maps (</em>.map), or configuration files containing credentials.

How it works

Many deployments put the web root directly inside a git repository checkout. If the web server serves all files in that directory, anyone can request /.git/config to find the remote repository URL, then use git clone to reconstruct the entire source tree including history.

Similarly, .env files store secrets injected at runtime (database passwords, API keys, JWT secrets) and are often accidentally placed in the web root.

Common exposed paths

  • /.git/config — Remote repo URL, branch names — allows full source reconstruction
  • /.env — DB passwords, API keys, JWT secrets
  • /.env.backup — Forgotten credential backups
  • /*.bak, /*.old — Backup copies of config files with secrets
  • /*.map — Source maps containing original pre-minified code
  • /swagger.json — API schema revealing internal endpoints
  • /phpinfo.php — PHP environment variables including secrets

Impact

  • Credential theft: DB passwords, API keys, cloud credentials
  • Full source code access: Business logic, auth bypass vulnerabilities
  • Internal infrastructure mapping: Hostnames, IP ranges, DB schemas

Defense

  • Configure the web server to deny access to .git/, .env, <em>.bak, </em>.map, *.config
  • Never deploy the application by serving the VCS working directory directly
  • Use secrets management (Vault, AWS Secrets Manager) — never store secrets in files within the web root
  • Run automated scanners (e.g., nikto, dirsearch) against your own app before attackers do

Tools used by attackers

  • dirsearch, ffuf, gobuster — wordlist-based path scanning
  • git-dumper — reconstructs a .git directory from an exposed web server
  • trufflehog — scans git history and files for leaked secrets
