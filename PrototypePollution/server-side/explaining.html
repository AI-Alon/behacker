Server-Side Prototype Pollution

What is it?

Server-side prototype pollution (SSPP) affects Node.js applications that use vulnerable merge/clone utilities on server-side data. Unlike client-side pollution (isolated to one browser tab), SSPP modifies Object.prototype for the entire Node.js process — affecting all subsequent requests from all users.

Why it's more severe than client-side

  • Persistence: Pollution survives across requests until the process restarts
  • All users affected: Every object created after pollution inherits the injected properties
  • RCE potential: Some Node.js internal modules read prototype properties

RCE via gadget chains

// POST /api/settings body:
{
  "__proto__": {
    "shell": "node",
    "NODE_OPTIONS": "--require /proc/self/fd/0"
  }
}

If a child_process.spawn() or template engine (Pug, EJS) later runs with these options — RCE.

Detection

Server-side pollution is harder to detect than client-side. Techniques:

  1. Response reflection: If {} is returned with unexpected properties → server prototype is polluted
  2. Error differences: Status code changes (e.g. 500 vs 200) can indicate prototype property access
  3. Timing: Polluting <em>_proto</em>_.toString with a slow function causes measurable delay

Known vulnerable packages

  • lodash < 4.17.17 (server-side _.merge)
  • hoek < 6.1.3 (Hapi.js utility)
  • flat < 5.0.1 (object flattener)
  • node-forge — various versions

Defense

// At application startup
Object.freeze(Object.prototype);

// In merge functions
const FORBIDDEN = new Set(['__proto__', 'constructor', 'prototype']);
function safeMerge(target, source) {
  for (const key of Object.keys(source)) {
    if (FORBIDDEN.has(key)) continue;
    // ...
  }
}

// Use JSON parse/stringify to strip prototype chains
const safe = JSON.parse(JSON.stringify(untrustedInput));
