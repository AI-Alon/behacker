SSRF Filter Bypass

What is it?

Developers often implement SSRF protection as a string-based blocklist checking for 127.0.0.1, localhost, and 169.254.169.254. These blocklists are trivially bypassable because there are dozens of equivalent representations for any IP address.

Bypass techniques

Numeric IP encoding

  • http://127.0.0.1/ — Loopback (blocked)
  • http://2130706433/ — Decimal: 127×16777216 + 1 = 2130706433
  • http://0x7f000001/ — Hexadecimal
  • http://0177.0.0.1/ — Octal first octet
  • http://127.1/ — Short-form (auto-expands)

IPv6

  • http://[::1]/ — IPv6 loopback, not matched by "127.0.0.1"
  • http://[::ffff:127.0.0.1]/ — IPv4-mapped IPv6

DNS-based bypasses

  • localtest.me — public DNS that always resolves to 127.0.0.1
  • nip.io — 127.0.0.1.nip.io resolves to 127.0.0.1
  • xip.io — similar service
  • Custom domain: attacker controls DNS → points to internal IP

Other techniques

  • Open redirect chains: allowlisted domain redirects to internal IP
  • DNS rebinding: domain resolves externally during allowlist check, then rebinds to internal IP
  • URL schemes: file://, dict://, gopher://, ldap://

Why blocklists fail

There is no finite list of representations for "127.0.0.1". Blocklists will always be incomplete.

Correct approach

  1. Resolve the URL to its final IP address using DNS lookup
  2. Check the resolved IP against blocked CIDR ranges
  3. Follow all redirects and re-check each destination
  4. Use an allowlist of permitted external domains/IPs
