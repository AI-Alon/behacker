Cloud Metadata SSRF

What is it?

Cloud instances (AWS EC2, GCP, Azure) expose a metadata API at the link-local address 169.254.169.254. This endpoint returns instance information, user-data scripts, and temporary IAM credentials — with no authentication, accessible only from within the instance. An SSRF vulnerability lets an external attacker reach this endpoint through the server.

AWS IMDSv1 critical endpoints

  • /latest/meta-data/iam/security-credentials/ — List of IAM role names
  • /latest/meta-data/iam/security-credentials/<role> — AccessKeyId, SecretAccessKey, Token
  • /latest/user-data — Bootstrap script (often contains secrets)
  • /latest/meta-data/hostname — Internal hostname

The Capital One breach (2019)

A WAF with SSRF allowed an attacker to:

  1. Call 169.254.169.254/latest/meta-data/iam/security-credentials/
  2. Retrieve IAM credentials for the instance's role
  3. Use credentials to list and download 100M+ customer records from S3
  4. Cost: $80M+ in fines and remediation

IMDSv2 (the fix)

IMDSv2 requires a two-step process:

# Step 1: Get a session token (requires PUT — browsers/simple SSRF can't do this)
TOKEN=$(curl -X PUT -H "X-aws-ec2-metadata-token-ttl-seconds: 21600" http://169.254.169.254/latest/api/token)

# Step 2: Use token in request
curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/

IMDSv2 blocks simple GET-based SSRF because it requires a PUT with a custom header first.

Additional defenses

  • Enforce IMDSv2 (disable IMDSv1) in all EC2 instance configurations
  • Apply least-privilege IAM roles — a web server shouldn't have s3:GetObject on all buckets
  • Use VPC endpoint policies and security groups to limit metadata access
  • Deploy WAF rules to block 169.254.169.254 in user-supplied URLs
