Race Condition (TOCTOU)

What is it?

A race condition (Time of Check / Time of Use) occurs when a security-relevant check and the subsequent action are not atomic. Multiple concurrent requests can all pass the check before any of them has completed the action — allowing the same resource to be consumed multiple times.

Classic pattern

Thread 1: CHECK balance ($50 >= $50) ✓
Thread 2: CHECK balance ($50 >= $50) ✓  ← races before T1 deducts
Thread 1: DEDUCT $50 → balance = $0
Thread 2: DEDUCT $50 → balance = -$50 ← double-spend!

Real-world examples

  • Gift card double-spend: Two simultaneous redemptions drain a card twice
  • Crypto exchanges: Withdraw the same funds to two addresses simultaneously
  • Coupon single-use bypass: Submit the same one-time coupon in parallel
  • Voting systems: Double-vote via concurrent API calls
  • CVE-2022-0492: Linux kernel cgroup race condition → privilege escalation

Attack execution

Tools like Burp Suite's Turbo Intruder, asyncio, or simple browser DevTools Network throttling can send HTTP requests with millisecond precision to hit the race window.

Defense

-- ✅ Atomic check-and-update in one statement
UPDATE accounts
SET balance = balance - ?
WHERE id = ? AND balance >= ?;
-- If 0 rows affected → insufficient funds (rejected atomically)

Or use database-level pessimistic locking:

BEGIN TRANSACTION;
SELECT balance FROM accounts WHERE id = ? FOR UPDATE;  -- locks row
-- check balance
UPDATE accounts SET balance = balance - ? WHERE id = ?;
COMMIT;

In application code: use mutex/semaphore per user ID, or idempotency keys that prevent the same logical operation from executing twice.
