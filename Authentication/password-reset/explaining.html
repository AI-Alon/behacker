Password Reset Poisoning — How It Works

Normal reset flow (safe)

The server generates a reset link using a hardcoded base URL from config:

const resetLink = `https://securecorp.com/reset?token=${token}`;

The domain is fixed — user input cannot influence it.

Vulnerable reset flow

The server trusts the Host header to build the reset URL:

const host      = req.headers['host'];          // ← attacker-controlled
const resetLink = `https://${host}/reset?token=${token}`;
sendEmail(user.email, resetLink);

The attack

  1. Attacker triggers a reset for the victim's account
  2. Attacker intercepts the request (Burp Suite) and changes the Host header:

```

Host: evil.com

```

  1. Server builds: https://evil.com/reset?token=a3f9c2e1...
  2. Victim receives a genuine email from no-reply@securecorp.com
  3. Victim clicks the link — request goes to evil.com with the token in the URL
  4. Attacker captures the token from their server logs
  5. Attacker visits https://securecorp.com/reset?token=a3f9c2e1... and resets the password

Variants

# Also check for:
X-Forwarded-Host: evil.com
X-Original-URL: https://evil.com/reset
X-Rewrite-URL: /reset

# Some servers trust these proxy headers over the real Host

The Fix

// Never build URLs from request headers

// Safe: read base URL from environment config
const BASE_URL   = process.env.APP_URL; // e.g. "https://securecorp.com"
const resetLink  = `${BASE_URL}/reset?token=${token}`;

// Also: validate the Host header against an allowlist if it must be used
const ALLOWED_HOSTS = ["securecorp.com", "www.securecorp.com"];
if (!ALLOWED_HOSTS.includes(req.headers.host)) {
  return res.status(400).send("Invalid host");
}

The domain used in reset emails must never be derived from user-supplied input.
