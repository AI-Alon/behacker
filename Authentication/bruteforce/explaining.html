Brute-Force Login — How It Works

What is it?

An attacker submits many username/password combinations until one works.

Unlike credential stuffing, the passwords are guesses — often from a wordlist of common passwords.

Why it works here

The server has no defences:

Attempt 001: admin:123456     → fail
Attempt 002: admin:password   → fail
Attempt 003: admin:qwerty     → fail
...
Attempt 014: admin:sunshine1  → SUCCESS

Each attempt gets an instant response. No delay, no lockout, no CAPTCHA.

Attack types

  • Dictionary attack: Try words from a wordlist (rockyou.txt, etc.)
  • Rule-based: Apply rules to words: password → P@ssw0rd
  • Mask attack: Try all combos matching a pattern: ?u?l?l?l?d?d
  • Credential stuffing: Use real leaked pairs (different attack — see that lab)

Real tools

# Hydra — network brute force
hydra -l admin -P rockyou.txt http-post-form "/login:user=^USER^&pass=^PASS^:Wrong"

# Burp Suite Intruder
# Load a wordlist as payload, target the password field

The Fix

// 1. Rate limiting — max N attempts per IP per minute
// 2. Account lockout — lock after 5 failures, require email unlock
// 3. Exponential back-off — each failure adds delay
// 4. CAPTCHA — require human verification after 3 failures
// 5. Strong password policy — minimum entropy, reject common passwords
// 6. MFA — even a correct password isn't enough alone

A properly defended login makes brute force computationally infeasible in any reasonable timeframe.
