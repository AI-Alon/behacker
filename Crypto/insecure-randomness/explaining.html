Insecure Randomness

What is it?

Cryptographic security requires randomness that is genuinely unpredictable. Using Math.random(), rand(), Random(), or timestamp-seeded PRNGs for security tokens (session IDs, password reset links, CSRF tokens, API keys) produces predictable output that attackers can reproduce.

Pseudorandom vs. cryptographically secure

  • Math.random() — Predictable (PRNG) — never use for security
  • new Random() in Java — Predictable (LCG) — never use for security
  • rand() in C — Predictable — never use for security
  • Date.now() as seed — Predictable (~millisecond range) — never use for security
  • crypto.getRandomValues() — Not predictable (CSPRNG) — safe for security
  • crypto.randomBytes() Node — Not predictable (OS entropy) — safe for security
  • /dev/urandom — Not predictable — safe for security

The attack: predicting a reset token

  1. Attacker requests a password reset for victim@example.com
  2. Server generates token: Math.floor(Math.random() * 0xFFFFFFFF).toString(16)
  3. Attacker notes the timestamp of the request
  4. Attacker brute-forces seeds near that timestamp (±5 seconds = 10,000 candidates)
  5. For each seed, generate the same token sequence → find the match
  6. Use the predicted token to reset the victim's password

With a 32-bit timestamp seed, the entire search space is ~4 billion values — trivially parallelizable.

Code comparison

Insecure:

// ❌ Predictable
const token = Math.random().toString(36).substring(2);
const resetToken = Date.now().toString(16);

Secure:

// ✅ Cryptographically secure (browser)
const bytes = new Uint8Array(32);
crypto.getRandomValues(bytes);
const token = Array.from(bytes).map(b => b.toString(16).padStart(2,'0')).join('');

// ✅ Cryptographically secure (Node.js)
import crypto from 'crypto';
const token = crypto.randomBytes(32).toString('hex');

Entropy requirements

  • Session token: minimum 128 bits (16 bytes)
  • Password reset token: minimum 128 bits (16 bytes)
  • CSRF token: minimum 128 bits (16 bytes)
  • API key: minimum 256 bits (32 bytes)

Real-world examples

  • PHP's mt_rand(): Used in early WordPress for reset tokens — predicted via seed recovery
  • Ruby's rand(): Without explicit seeding, time-based — exploited in several gems
  • Java's java.util.Random: Linear congruential generator — state recoverable after observing 2 outputs
